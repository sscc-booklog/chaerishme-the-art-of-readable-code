## Part 3 : 코드 재작성하기

### 10장 상관없는 하위 문제 추출하기


**엔지니어링**
- 큰 문제 → 작은 문제로 분해 
- 각 문제에 대한 해결책 구하기
- 하나의 해결책으로 맞추기

즉, 큰 흐름과 관계가 적은 하위문제를 적극적으로 발견해서 추출하기 <br>
→ 이는 다음의 순서를 따른다.

1. 상위 수준에서 본 이 코드의 목적은 무엇인가?
2. 코드의 모든 줄 : 직접적으로 목적을 위해 존재하는가? 혹은 목적 자체와 직접적으로는 상관없는 하위 문제를 해결하는가? 
3. 원래 목적과 직접적으로 관련되지 않은 하위 문제를 해결하는 코드가 많으면 이를 추출해서 별도의 함수로 만들기


<br>

> 이러한 기법을 사용하는 방법 

- 순수한 유틸리티 코드
- 일반적인 목적의 코드에서 함수 추출 <br>
→ 함수를 호출하는 코드를 간단히 만들고, 다른 곳에서도 간편히 사용 가능. 
손쉽게 개선 가능.
- 일반적인 목적을 가진 코드를 많이 만들어라
- 특정한 프로젝트를 위한 기능
- 기존의 인터페이스를 단순화하기
- 자신의 필요에 맞춰서 인터페이스의 형태를 바꾸기

단, 너무 지나치게 추출하지 말 것.

<br>

### 요약
**일반적인 목적의 코드를 프로젝트의 특정 코드에서 분리하라**



<br>

# insight 💡

이번 내용을 공부하면서 코드를 잘 만든다는 건 단순히 기능을 구현하는 게 아니라 ‘문제를 제대로 쪼개는 능력’이라는 걸 다시 느꼈다.
특히 내가 평소에 함수 하나에 여러 일을 욱여넣는 경향이 있다는 걸 인정하게 됐다.

하위 문제를 분리해보니, 원래 코드가 무엇을 “진짜로” 하려고 하는지 더 선명해지고, 부수적인 처리들은 함수 바깥으로 빼는 순간 코드 흐름이 훨씬 깔끔해졌다.

또, 이렇게 추출한 일반적인 목적의 코드들은 프로젝트를 넘어서 재사용 가능한 자산이 된다는 점이다.
즉, ‘지금 당장의 편함’보다 ‘앞으로의 확장성’을 생각하며 코드를 구성해야 한다는 걸 깨달았다.

무엇보다 중요한 건 함수 추출이 목적이 아니라 기준이라는 것. “이 줄이 정말로 이 함수의 목적에 기여하는가?” 이 질문을 기준으로 분리 여부를 판단해야 한다는 걸 배웠다.

→ 코드를 잘 만든다는 건 ‘문제의 본질을 남기고, 본질과 상관없는 것들을 분리하는 능력’이다.
