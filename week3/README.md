## summary 📖


### 3장 오해할 수 없는 이름들

본인이 지은 이름을 *다른 사람들이 다른 의미로 해석할 수 있을까?* 라는 질문을 던져보며 철저하게 확인해야 한다.
즉, 최악의 경우를 가정하고 이름의 의미가 잘못 이해되는 가능성을 고려해야 한다.
<br>


- 상한과 하한을 정할 때는 **min, max**
- 경계를 포함한다면 **first, last**
- 경계의 시작만 포함하고 마지막은 배제한다면 **begin, end**
- 불리언값 이름을 정할 때는 **is, has** 사용. 부정의 의미는 피하기 
- 사람들이 **일반적으로 생각하는 사실**에 유의 (get(), size()의 간단 동작 등)


### 4장 미학

- 코드를 읽는 사람에게 친숙한, 일관성 있는 레이아웃 사용
- 비슷한 코드는 비슷해보이게 만들기
- 서로 연관된 코드는 하나의 블록으로 묶기
-  열을 맞춰 코딩하기
```
public class PerformanceTester{
    //  TcpConnectionSimulator (처리량, 지연속도, 흔들림, 패킷_손실)
    //                         [Kbps]  [ms]   [ms]  [percent]

    public static final TcpConnectionSimulator wifi = 
        new TcpConnectionSimulator(500,    80,  200,    1);

    public static final TcpConnectionSimulator t3_fiber = 
        new TcpConnectionSimulator(45000,  10,    0,    0);
}
```

- 헬퍼 메소드 사용하여 가독성 향상시키기
- 의미있는 순서 사용
- 선언문 블록으로 구성하기
- 코드를 문단으로 쪼개기


### 5장 주석에 담아야 하는 대상
> 설명하지 말아야 하는 것

- 코드에서 빠르게 유추할 수 있는 내용
- 설명 자체를 위한 설명
- 나쁜 코드에 변명을 위해 주석 달기

> 생각을 기록하기

- 중요한 통잘을 기록하기
- 코드에 있는 결함 설명하기
- 상수에 대한 상세한 설명

> 코드를 읽는 사람의 입장이 되어라

- 나올 것 같은 질문 예측하기
- 큰 그림에 대한 주석 달기 (상위수준/파일수준)
- 함수를 요약하는 주석

## insight 💡
코드 내에 열을 사용하여 정렬할 생각은 하지 못했는데, 예시를 보니 열을 사용하는 것이 직관적으로 이해가 되었다. 또한 변수명을 지을 때 깊게 고려하지 않고 지었는데, 책에서는 깊은 고려 후 변수명을 짓는 것 같아 내 작명 센스를 되돌아보게 되었다...

코딩 또한 작문과 다를 바가 없는 듯하다. 코딩하다가 정신을 놓게 될 때가 많은데, 앞으로는 정신을 붙잡고 생각하며 코딩함과 동시에 주석도 신경써서 달아야겠다.
